01 · WHAT CAP-EVOLVE IS

Improve agentic capabilities iteratively — with agents

Agents drive both the outer search loop and the inner optimization step. Bring the agent and the eval you already have; cap-evolve returns gate-verified improvements measured on held-out splits.

Capabilities we optimize

System prompts Tool code Skill packages MCP surfaces

Pick one — or combine, e.g. [system-prompt, tools].

Frame cap-evolve as an iterative loop where agents drive both the outer search (which capability to change next) and the inner optimization (what edit to propose). No training compute, no model risk. The 4 surfaces chips open the door for cross-team applicability.
01 · WHAT CAP-EVOLVE IS

Why cap-evolve is different

Six design choices that separate it from prior tool-optimization work.

01
Agents optimize agents
The optimizer is a coding-agent CLI. No gradients, no heuristic search.
02
Any agentic capability
Prompts, tool code, MCP surfaces, skill packages — one or combined.
03
Optimization under constraints
Cost-capped iterations. Wall-clock next.
04
Flexible intake
One-prompt onboarding. Agent-driven wiring, dynamic config.
05
Flexible state
Git-committed markdown. Browseable, editable, resumable.
06
Rigorous evaluation
Sealed test, gate-verified acceptance, artifacts committed with every result.
Six differentiators. #3 Flexible intake is new here (added to the site tile set too): one-prompt onboarding — cap-evolve interviews you, installs the benchmark, wires the adapter, configures per capability. #4 budget-caps and #5 resumable state are still firming up.

3.4Memory · KB · LLM-wiki

State / memory / knowledge base / LLM-wiki — one cross-run substrate that IS the org's living record of how agents get better.

What ships today

  • Per-run markdown state — LEDGER · JOURNAL · PROCESS · RUNMAP
  • Every iteration is a git commit
  • Optimizer receives per-task IMPACT + protect-set each iteration
  • Rejected candidates never re-proposed within a run

The stretch — a durable KB

  • Cross-run learnings — what worked / regressed across many runs
  • LLM-queryable substrate — an MCP / wiki agents can read at plan time
  • First-class rationale — decisions and their why, not just diffs
  • Portable — export between org units and benchmarks
Why it's strategic
Today, cap-evolve's "memory" is a run artifact. Tomorrow, it's an evolving org-wide substrate — every run adds to a shared KB that every future agent reads. A real moat.

3.5Simulator from traces

Reconstruct enough of the tool environment from recorded I/O to re-execute agents — closed-loop, no live systems.

The mechanism

  • Every tool call in a trace has captured inputs + outputs
  • Read-only tools (query_*, fetch_*) simulate cheaply
  • Reconstruct a frozen replica; agent re-executes against it

Why it matters for cap-evolve

  • Score candidate skills live — not just recorded trajectories
  • No risk to live customer systems (Parsec fit)
  • Cheaper eval loop → more iterations per dollar

Status

  • Separate IBM team already building trace-based simulators
  • Natural collaboration with cap-evolve — clean integration point
  • Scoped as Parsec phase-2 extension, not a prerequisite

Reference

rhdp-parsec/docs/rh_collaboration_proposal.md § 7 — Optional collaboration extension: simulator from traces
